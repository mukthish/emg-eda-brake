## Requirements, Classification, Rationale, and Testability


| Requirement ID | Requirement (single declarative sentence) | Classification (Functional / Safety / Non-Functional) | Rationale (Why important? What could go wrong if violated?) | Test Intent / Testability (How violation would be detected) |
|---------------|-------------------------------------------|--------------------------------------------------------|-------------------------------------------------------------|-------------------------------------------------------------|
| R-1 | The system shall treat EDA as optional input and shall remain operational when EDA samples are absent for up to 60 seconds. | Safety | Missing EDA must not disable braking intent detection. | Run EMG-only stream for 60s and verify no fault and no blockage of detection. |
| R-2 | The system shall detect sustained loss of valid input data if no valid sample is received for 750 ms. | Safety | Stale data could result in incorrect intent decisions and unsafe requests. | Stop input and verify fault is set at ~750ms and output stays OFF. |
| R-3 | The system shall assert the emergency brake output when the EMG signal envelope exceeds the configured threshold for more than 150ms.| Functional | To detect driver braking intent early. If violated, the system fails its primary purpose of providing emergency safety | Violation is detected if the brake output remains low while the input EMG signal is sustained above the threshold for >150ms. |
| R-4 | The system shall transition to a safe (non-actuating) state if valid input data is not received for more than 500ms. | Safety | To prevent erratic behavior during sensor disconnection or signal loss. If violated, the system might act on stale noise. | Violation is detected if the system remains in an active/braking state after the input data stream is disconnected for >500ms. |
| R-5 | The system shall accept physiological data input from both simulated sources and live sensor hardware without changing the core detection logic. | Non-Functional | To ensure the system can be developed using simulations and later integrated with real sensors. If violated, code must be rewritten for integration, risking bugs. | Violation is detected if switching between the simulation source and the live sensor requires recompiling or modifying the intent detection module. |
| R-6 | The system shall generate the brake request signal within 50ms of the braking intent condition being met. | Non-Functional | To ensure the electronic brake triggers faster than a human mechanical reaction. If violated, the safety benefit of "early detection" is lost. | Violation is detected if the latency between the "intent detected" event and the brake signal activation exceeds 20ms. |
| R-7 | The system shall transition to a non-asserting safe state upon detection of internal software execution failures. | Safety | Prevents the system from "freezing" in an active braking state or becoming unresponsive during a crash. | Violation is detected if a software fault (e.g., forced infinite loop) occurs and the brake signal remains active. |
| R-8 | The system shall be ready to perform intent detection within 500ms of a power-on or system reset. | Non-Functional | In a vehicle, safety systems must be active immediately upon startup; a long "boot time" leaves the driver unprotected. | Test: Power-cycle the device and verify it can process a trigger signal within the first half-second. |
| R-9 | The system shall perform intent detection at a minimum rate of 50 Hz. | Non-Functional | Timely detection is required for early braking assistance. | Violation is detected if evaluation runs slower than once per 20 ms. |
| R-10 | The system shall ignore isolated rapid EMG spikes that do not persist beyond a single sampling interval Safety| Safety | A single-sample EMG spike is usually caused by foot adjustment, sensor movement, or random muscle twitch rather than genuine braking intent. | Violation is detected if a single-sample spike triggers braking intent. |
